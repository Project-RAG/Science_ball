# Правила контроля качества

## 1. Предотвращение расширения объёма DeepSeek

Давайте DeepSeek ровно один файл задачи.

Промпт должен содержать:
- реализовать только эту задачу;
- не реализовывать будущие задачи;
- не переписывать архитектуру;
- не редактировать `docs/SDD.md`;
- перечислить изменённые файлы;
- перечислить заглушки и проверки.

Отклонять вывод DeepSeek, если он:
- создаёт большие несвязанные модули;
- реализует несколько этапов;
- хардкодит секреты;
- вызывает YandexGPT напрямую вне `LLMGateway`;
- помещает логику хранилищ в routes;
- изменяет SDD или документы оркестрации без запроса.

---

## 2. Предотвращение переписывания проекта Gemma

Промпт Gemma должен содержать:
- интегрировать, не переписывать;
- сохранять существующую архитектуру;
- исправлять только то, что нужно для текущей задачи;
- записывать более широкие исправления в handoff;
- не удалять предыдущую работу без необходимости и документирования.

Отклонять вывод Gemma, если он:
- заменяет структуру каталогов без причины;
- изменяет технологический выбор из SDD;
- объединяет будущие задачи с текущей;
- удаляет тесты вместо их исправления;
- скрывает неудачные проверки.

---

## 3. Обработка неработающего сгенерированного кода

Если код DeepSeek не работает:

1. Gemma находит наименьшую поверхность отказа.
2. Gemma сначала исправляет импорты/конфигурацию/зависимости.
3. Gemma избегает перепроектирования, если сгенерированный код не противоречит SDD.
4. Если всё ещё не работает, Gemma документирует проблему в `Known Issues`.
5. Задача не коммитится, если отказ не неблокирующий и явно задокументирован.

---

## 4. Политика заглушек и моков

Заглушки/моки разрешены только когда:
- реальная интеграция запланирована на более позднюю задачу;
- заглушка имеет понятный интерфейс, совместимый с планируемой реализацией;
- заглушка задокументирована в `docs/HANDOFF.md`.

Каждая запись заглушки должна включать:
- файл/модуль;
- поведение;
- причину;
- задачу, в которой она должна быть заменена.

Никогда не выдавать заглушку за production-ready поведение.

---

## 5. Политика известных проблем

Известные проблемы должны отслеживаться в `docs/HANDOFF.md`.

Каждая проблема должна включать:
- ID;
- краткое описание;
- серьёзность;
- обходное решение;
- целевую задачу или точку принятия решения.

Уровни серьёзности:
- blocker — невозможно продолжить или коммитить;
- high — задача работает, но следующий этап рискован;
- medium — приемлемо для MVP, если отслеживается;
- low — чистка/документация.

---

## 6. Чеклист проверки секретов

Перед коммитом проверьте:

```bash
git diff
```

Ищите:
- `YANDEX_API_KEY` с реальным значением;
- пароли;
- токены;
- приватные ключи;
- заголовки Authorization;
- `.env`, добавленный в git.

`.env.example` должен содержать только заполнители.

---

## 7. Архитектурный чеклист

Для каждой задачи проверьте:

- FastAPI routes тонкие.
- Бизнес-логика в сервисах.
- PostgreSQL persistence в моделях/репозиториях.
- Доступ к Elasticsearch в клиенте/сервисе.
- Доступ к Neo4j в графовом клиенте/сервисе.
- Доступ к MinIO в клиенте объектного хранилища.
- Доступ к LLM только через `LLMGateway`.
- Поиск всегда учитывает `access_level` после реализации поиска.
- Факты трассируются к документу/чанку после реализации фактов.

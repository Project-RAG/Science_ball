# Правила AI для R&D Knowledge Map

## 1. Назначение

Данный документ определяет правила для AI-ассистированной реализации проекта R&D Knowledge Map.

Основные источники истины:
- `docs/SDD.md` — архитектура и требования к продукту. Не редактировать во время задач реализации.
- `docs/IMPLEMENTATION_PLAN.md` — поэтапный план выполнения.
- `docs/HANDOFF.md` — текущее состояние реализации.
- `docs/tasks/TASK_XXX_...md` — текущая ограниченная задача.

Проект должен реализовываться итеративно, небольшими верифицируемыми задачами. Ни один AI-агент не может генерировать или переписывать весь проект сразу.

---

## 2. Роли агентов

### 2.1 DeepSeek в Kodik

DeepSeek — генератор кода для одной небольшой задачи за раз.

DeepSeek должен:
- читать только релевантные документы проекта и текущую задачу;
- реализовывать минимум кода, требуемый текущей задачей;
- держать бизнес-логику вне API routes;
- размещать доступ к внешним хранилищам за клиентами/репозиториями;
- использовать заглушки/моки только когда это явно разрешено задачей;
- перечислять все созданные/изменённые файлы в конце;
- перечислять команды для валидации результата.

DeepSeek не должен:
- реализовывать будущие задачи;
- переписывать архитектуру из `docs/SDD.md`;
- редактировать `docs/SDD.md`;
- делать коммиты;
- хранить секреты в исходном коде, frontend, тестах, документах, Docker-образах или git;
- вызывать YandexGPT напрямую из routes или бизнес-сервисов;
- помещать логику PostgreSQL/Elasticsearch/Neo4j/MinIO/Redis напрямую в routes.

### 2.2 Gemma-4-31B в Claude Code

Gemma — интегратор. Берёт результат DeepSeek и делает его работоспособным в реальном репозитории.

Gemma должен:
- проверять, что изменения соответствуют текущей задаче;
- исправлять импорты, зависимости, конфигурацию, Docker, Alembic, тесты и форматирование;
- сохранять SDD-архитектуру;
- запускать целенаправленные проверки;
- обновлять `docs/HANDOFF.md` после каждой задачи;
- записывать заглушки, моки, известные проблемы и следующий шаг;
- сохранять изменения минимальными.

Gemma не должен:
- переписывать проект с нуля;
- заменять архитектуру или технологии хранения;
- редактировать `docs/SDD.md`;
- реализовывать несколько задач за один проход;
- удалять результаты предыдущих задач, если они не сломаны и исправление не задокументировано;
- делать коммит без явного указания пользователя.

---

## 3. Контроль объёма

Каждая задача ограничена файлами и поведением, указанными в её файле.

Допустимые изменения вне рамок:
- исправления импортов;
- обновление списка зависимостей;
- минимальная конфигурационная привязка;
- тесты, необходимые для валидации задачи;
- обновление `docs/HANDOFF.md` от Gemma.

Если агенту нужны более широкие изменения, он должен записать:
- почему изменение было необходимо;
- какие файлы были затронуты;
- создаёт ли это новую известную проблему или открытый вопрос.

Запрещённое расширение объёма:
- реализация будущих API раньше времени;
- добавление frontend во время backend-only задач;
- добавление LLM-провайдера до задачи LLM;
- добавление продвинутого NLP до задач MVP со словарями/regex;
- добавление стека мониторинга до того, как MVP его потребует.

---

## 4. Секреты и безопасность

Секреты должны существовать только в локальном `.env` и переменных окружения времени выполнения.

Никогда не храните в git:
- `YANDEX_API_KEY`;
- реальный `YANDEX_FOLDER_ID`, если он приватный;
- пароли PostgreSQL;
- пароли Neo4j;
- ключи доступа/секреты MinIO;
- JWT-секреты;
- облачные учётные данные;
- приватные ключи;
- токены.

`.env.example` может содержать только заполнители вроде `change_me`, никогда реальные значения.

Frontend никогда не должен получать или ссылаться на Yandex API-ключи. Все LLM-вызовы идут через backend-only `LLMGateway`.

Логи не должны включать:
- API-ключи;
- заголовки авторизации;
- сырые секреты;
- полные промпты, содержащие конфиденциальный/ограниченный текст документов.

Для ограниченных документов отправляйте в LLM только извлечённые факты/доказательства, а не полный текст документа.

---

## 5. Требования к работоспособности

Каждая задача должна оставлять репозиторий в максимально возможном рабочем состоянии.

Минимальные ожидания:
- приложение импортируется успешно;
- базовый health endpoint работает после задачи скелета backend;
- Docker Compose config валидируется после введения Docker;
- тесты реализованной логики проходят;
- миграции генерируются и выполняются после введения Alembic;
- отказавшие внешние сервисы явно деградируют, если задача допускает заглушки.

Если задача ещё не может быть полностью работоспособной, это должно быть задокументировано в `docs/HANDOFF.md` в разделах `Known Issues` и `Stubs/Mocks`.

---

## 6. Правила Docker

Docker Compose для MVP должен включать, когда введён:
- backend;
- worker;
- postgres;
- elasticsearch;
- neo4j;
- redis;
- minio.

Опциональные сервисы откладываются, если задача явно их не запрашивает:
- frontend;
- ollama;
- prometheus;
- grafana;
- kibana.

Правила Docker:
- не встраивать секреты в образы;
- передавать секреты через переменные окружения из `.env`;
- держать `.env.example` безопасным;
- использовать healthchecks, где это практично;
- backend и worker должны использовать одну модель настроек;
- изменения Docker должны валидироваться командой `docker compose config`, когда Docker доступен.

---

## 7. Правила окружения и настроек

Настройки должны быть централизованы в конфигурации backend, предпочтительно в `backend/app/settings.py`.

Обязательные группы конфигурации со временем:
- настройки приложения;
- URL PostgreSQL;
- URL Redis;
- URL Elasticsearch;
- URI Neo4j и учётные данные;
- endpoint MinIO и учётные данные;
- настройки LLM-провайдера;
- настройки YandexGPT;
- настройки локального LLM fallback.

Значения по умолчанию могут быть удобными для разработки, но секреты должны оставаться заполнителями.

---

## 8. Правила репозитория и слоёв

Routes должны быть тонкими.

Рекомендуемая слоистая архитектура backend:
- `api/routes/*` — только HTTP-граница;
- `schemas/*` — DTO запросов/ответов;
- `services/*` — бизнес-процессы;
- `repositories/*` — транзакционные операции с БД;
- `db/*` — внешние клиенты/фабрики сессий;
- `models/*` — SQLAlchemy модели;
- `workers/*` — Celery задачи.

Внешние системы не должны вызываться напрямую из routes:
- доступ к PostgreSQL через сессии/репозитории;
- доступ к Elasticsearch через клиент/сервис;
- доступ к Neo4j через графовый клиент/сервис;
- доступ к MinIO через клиент объектного хранилища;
- доступ к Redis/Celery через worker/config абстракции;
- доступ к LLM через `LLMGateway`.

---

## 9. Правила PostgreSQL

PostgreSQL — транзакционный источник истины для:
- пользователей и ролей;
- документов;
- задач приёма (ingestion jobs);
- метаданных чанков;
- фактов;
- версий фактов;
- проверок (reviews);
- журнала аудита.

Правила:
- использовать SQLAlchemy модели;
- использовать Alembic миграции;
- не хранить исходные файлы в PostgreSQL;
- строки документов должны включать `access_level`;
- факты должны включать `source_document_id` и желательно `source_chunk_id`;
- изменения фактов должны создавать версии в `fact_versions` после реализации редактирования фактов;
- не удалять историю незаметно.

---

## 10. Правила Elasticsearch

Elasticsearch — основной поисковый слой для:
- полнотекстового поиска;
- векторного поиска;
- числовых фильтров;
- извлечения доказательств.

Правила:
- все индексируемые документы/чанки/факты должны включать `access_level`;
- каждый поисковый запрос должен применять фильтр доступа;
- маппинги должны использовать подходящие типы: `keyword`, `text`, `dense_vector`, `float`, `integer`, `date`, `nested`;
- не смешивать эмбеддинги разных размерностей в одном индексе;
- если размерность эмбеддингов меняется, создавать новую версию индекса;
- не заменять Elasticsearch на PostgreSQL search для MVP-функций поиска.

---

## 11. Правила Neo4j

Neo4j используется для графовых обходов и визуализации.

Правила:
- не заменять Neo4j на Elasticsearch или PostgreSQL для эмуляции графа;
- использовать ограничения (constraints) для стабильных ID узлов;
- связи, созданные из фактов, должны содержать метаданные трассировки, где возможно:
  - `fact_id`;
  - `source_document_id`;
  - `confidence`;
  - `verification_status`;
  - временные метки.

---

## 12. Правила MinIO

MinIO хранит:
- исходные загруженные документы;
- распарсенные артефакты;
- экспорты.

Правила:
- не хранить исходные бинарные документы в PostgreSQL;
- ключи объектов должны быть достаточно детерминированными, чтобы избежать коллизий;
- метаданные документа в PostgreSQL должны указывать на bucket/object key в MinIO;
- API загрузки должны проверять тип и размер файла при реализации;
- локальная разработка может автоматически создавать требуемые bucket'ы.

---

## 13. Правила Redis и Celery

Redis используется для очередей, кэша и поддержки статусов задач.

Celery используется для фоновых задач.

Правила:
- ingestion должен стать асинхронным после того, как базовая загрузка заработает;
- routes должны ставить задачи в очередь вместо долгой inline-обработки;
- статус задачи должен быть виден через endpoint'ы ingestion job;
- упавшие задачи должны сохранять информацию об ошибке без утечки секретов.

---

## 14. Правила LLMGateway

Весь доступ к LLM должен идти через:

```text
backend/app/services/llm/llm_gateway.py
```

Структура провайдеров:

```text
LLMProvider
  YandexLLMProvider
  LocalLLMProvider
  MockLLMProvider
```

Правила:
- бизнес-логика не должна вызывать Yandex API напрямую;
- routes не должны вызывать Yandex API напрямую;
- тесты должны использовать `MockLLMProvider`;
- YandexGPT не является источником истины;
- LLM должна отвечать только на основании доказательств;
- результаты query understanding должны валидироваться через Pydantic;
- если YandexGPT падает, использовать детерминированный fallback;
- не логировать API-ключи и заголовки авторизации.

---

## 15. Правила извлечения числовых данных

Числа должны извлекаться детерминированно с помощью regex/parser-логики.

Правила:
- не полагаться на LLM как единственный источник точных числовых значений;
- нормализовать единицы измерения;
- сохранять сырой текстовый диапазон, когда возможно;
- числовые факты должны ссылаться на `document_id` и `chunk_id`, когда доступны;
- факты, извлечённые только LLM, не могут получать confidence выше `0.65` без верификации.

---

## 16. Правила тестирования

Каждая задача должна добавлять или обновлять целенаправленные тесты, когда это возможно.

Обязательные области тестирования со временем:
- health endpoint;
- загрузка настроек;
- инициализация клиентов хранилищ с моками;
- SQLAlchemy модели и проверка Alembic миграций;
- загрузка документов с мокированным MinIO;
- парсеры;
- разбиение на чанки;
- числовое извлечение;
- нормализация единиц;
- fallback query understanding;
- LLMGateway с mock-провайдером;
- фильтрация доступа в поисковых запросах.

Не используйте реальный YandexGPT в тестах.

---

## 17. Правила handoff

Gemma должен обновлять `docs/HANDOFF.md` после каждой задачи:
- ID и название задачи;
- статус;
- изменённые файлы;
- команды валидации и результаты;
- заглушки/моки;
- известные проблемы;
- открытые вопросы;
- рекомендация следующей задачи;
- безопасность коммита.

---

## 18. Правила Git

Одна задача обычно даёт один коммит.

Коммитить только когда:
- Definition of Done задачи выполнен;
- проверки проходят или приемлемые ограничения задокументированы;
- `docs/HANDOFF.md` обновлён;
- diff не содержит секретов;
- diff не содержит масштабных несвязанных переписываний.

Рекомендуемое сообщение коммита:

```text
TASK_XXX: краткое императивное описание
